## 一、分支模型选择

### 1.1 推荐方案：简化 Git Flow

适合大多数国内中小团队的实际情况：

```
main     ──●──────●──────●──── 生产环境（受保护）
            \    / \    /
dev      ──●──●─●──●──●─●──── 开发/测试环境
             \  /    \  /
feat/x       ●●      ●●       功能分支
```

### 1.2 分支说明

| 分支 | 用途 | 保护级别 | 创建来源 | 合并目标 |
|------|------|---------|----------|----------|
| `main` | 生产环境代码 | 高 | - | - |
| `dev` | 开发/测试环境 | 中 | main | main |
| `feat/*` | 功能分支 | 无 | dev | dev |
| `fix/*` | 修复分支 | 无 | dev | dev |
| `hotfix/*` | 紧急修复 | 无 | main | main + dev |
| `release/*` | 发布分支 | 中 | dev | main + dev |

### 1.3 分支规则

- `main` 分支受保护，只能通过 PR/MR 合并
- `dev` 分支对应测试环境，自动部署
- 功能分支从 `dev` 拉出，合回 `dev`
- `dev` 测试通过后，合并到 `main` 进行发布

## 二、分支命名规范

### 2.1 命名格式

**统一格式**：`{类型}/{描述}`

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat/user-login` |
| `fix` | Bug 修复 | `fix/login-timeout` |
| `hotfix` | 紧急修复 | `hotfix/critical-bug` |
| `release` | 版本发布 | `release/v1.0.0` |
| `docs` | 文档更新 | `docs/api-update` |
| `refactor` | 重构 | `refactor/user-service` |
| `test` | 测试相关 | `test/unit-tests` |

### 2.2 命名示例

```bash
# 功能分支
feat/user-login              # 用户登录功能
feat/booking-create          # 创建预约功能
feat/TAPD-1234-order-refund  # 关联任务编号

# 修复分支
fix/payment-callback         # 支付回调修复
fix/JIRA-5678-null-pointer   # 关联 Bug 编号

# 紧急修复
hotfix/v2.0.1                # 线上紧急修复
hotfix/fix-login-crash       # 描述性命名

# 发布分支
release/v1.0.0               # 版本发布
release/2024-03-sprint       # 按迭代命名

# 其他分支
docs/api-update              # 文档更新
refactor/user-service        # 重构用户服务
test/unit-tests              # 单元测试
```

### 2.3 命名规则

1. 全部小写，用 `-` 连接单词（不用下划线或驼峰）
2. 前缀明确分支类型：`feat/`、`fix/`、`hotfix/`、`release/`
3. 关联任务管理平台的编号（如有）：`feat/TAPD-12345-description`
4. 长度适中，能看出分支目的即可

## 三、分支工作流程

### 3.1 功能开发流程

```bash
# 1. 切换到 dev 分支并更新
git checkout dev
git pull origin dev

# 2. 创建功能分支
git checkout -b feat/user-login

# 3. 开发完成后提交
git add .
git commit -m "feat(用户): 实现微信登录功能"

# 4. 推送到远程
git push origin feat/user-login

# 5. 发起 Pull Request 合并到 dev
# 在 Git 平台（Gitee/Coding/GitLab）上操作

# 6. 合并后删除本地分支
git checkout dev
git pull origin dev
git branch -d feat/user-login

# 7. 删除远程分支
git push origin --delete feat/user-login
```

### 3.2 Bug 修复流程

```bash
# 1. 从 dev 创建修复分支
git checkout dev
git pull origin dev
git checkout -b fix/login-timeout

# 2. 修复并提交
git add .
git commit -m "fix(登录): 修复登录超时问题"

# 3. 推送到远程并创建 PR
git push origin fix/login-timeout

# 4. 合并后清理分支
git checkout dev
git pull origin dev
git branch -d fix/login-timeout
git push origin --delete fix/login-timeout
```

### 3.3 紧急修复流程（Hotfix）

```bash
# 1. 从 main 创建紧急修复分支
git checkout main
git pull origin main
git checkout -b hotfix/critical-bug

# 2. 修复并提交
git add .
git commit -m "fix(全局): 修复严重安全漏洞"

# 3. 推送到远程并创建 PR 到 main
git push origin hotfix/critical-bug

# 4. 合并到 main 后，同步到 dev
git checkout main
git pull origin main
git checkout dev
git pull origin dev
git merge main
git push origin dev

# 5. 清理分支
git branch -d hotfix/critical-bug
git push origin --delete hotfix/critical-bug
```

### 3.4 版本发布流程

```bash
# 1. 从 dev 创建发布分支
git checkout dev
git pull origin dev
git checkout -b release/v1.0.0

# 2. 更新版本号（package.json、app.json 等）
# 3. 修复测试中发现的问题（只修 Bug，不加功能）
git add .
git commit -m "chore(release): 准备 v1.0.0 发布"

# 4. 合并到 main
git checkout main
git pull origin main
git merge release/v1.0.0
git tag -a v1.0.0 -m "发布 v1.0.0"
git push origin main --tags

# 5. 合并回 dev
git checkout dev
git pull origin dev
git merge release/v1.0.0
git push origin dev

# 6. 删除发布分支
git branch -d release/v1.0.0
git push origin --delete release/v1.0.0
```

## 四、Pull Request 规范

### 4.1 PR 创建规范

#### PR 标题格式

```
[类型](范围): 简要描述

示例：
feat(用户): 添加微信一键登录功能
fix(预约): 修复时间冲突检测逻辑
```

#### PR 描述模板

```markdown
## 变更说明
<!-- 简要描述这次改动做了什么，解决了什么问题 -->

## 变更类型
- [ ] 新功能（feat）
- [ ] Bug 修复（fix）
- [ ] 重构（refactor）
- [ ] 性能优化（perf）
- [ ] 文档更新（docs）
- [ ] 其他：

## 关联信息
- 需求/Bug 链接：
- 设计文档：

## 改动范围
<!-- 列出主要改动的模块和文件 -->
- 
- 
- 

## 测试情况
- [ ] 单元测试通过
- [ ] 手动测试通过
- [ ] 相关模块回归测试通过

## 测试方法
<!-- 描述如何验证这次改动 -->

## 影响范围
<!-- 这次改动可能影响哪些功能？是否需要通知其他团队？ -->

## 部署注意事项
- [ ] 需要执行数据库迁移
- [ ] 需要更新配置文件
- [ ] 需要更新环境变量
- [ ] 无特殊注意事项

## 截图/录屏
<!-- 如果涉及 UI 变更，贴截图或录屏 -->
```

### 4.2 PR 审查流程

#### 审查要求

| 项目 | 要求 |
|------|------|
| 审查人 | 至少 1 名其他团队成员 |
| 审查内容 | 代码规范、逻辑正确性、安全性、性能 |
| 审查方式 | Pull Request + 代码评论 |
| 合并条件 | 所有评论已解决 + 审查人批准 + CI 通过 |

#### 审查检查清单

```markdown
## 代码审查检查清单

### 规范性检查
- [ ] 代码符合命名规范
- [ ] 必要的注释已添加
- [ ] 无 console.log 等调试代码残留
- [ ] 无硬编码的敏感信息

### 功能性检查
- [ ] 功能实现符合需求
- [ ] 边界条件已处理
- [ ] 错误处理已完善

### 安全性检查
- [ ] 用户输入已验证
- [ ] 敏感数据已加密
- [ ] 权限控制已正确实现

### 性能检查
- [ ] 无明显的性能问题
- [ ] 大数据量已考虑分页
- [ ] 图片等资源已优化
```

## 五、版本控制标准

### 5.1 版本号规范

采用**语义化版本控制（Semantic Versioning）**：

```
版本格式：主版本号.次版本号.修订号
示例：1.0.0

主版本号：重大功能更新，不兼容的 API 修改
次版本号：向下兼容的功能新增
修订号：向下兼容的问题修复
```

### 5.2 版本发布流程

```bash
# 1. 创建发布分支
git checkout dev
git checkout -b release/v1.0.0

# 2. 更新版本号
# 更新 package.json、app.json 等文件中的版本号

# 3. 修复测试中发现的问题
# 只修 Bug，不加功能

# 4. 合并到 main
git checkout main
git merge release/v1.0.0
git tag -a v1.0.0 -m "发布 v1.0.0"
git push origin main --tags

# 5. 合并回 dev
git checkout dev
git merge release/v1.0.0
git push origin dev

# 6. 删除发布分支
git branch -d release/v1.0.0
```

### 5.3 版本标签

```bash
# 创建标签
git tag -a v1.0.0 -m "发布 v1.0.0"

# 推送标签到远程
git push origin v1.0.0

# 推送所有标签
git push origin --tags

# 删除本地标签
git tag -d v1.0.0

# 删除远程标签
git push origin --delete v1.0.0
```

## 六、常用 Git 命令

### 6.1 分支操作

```bash
# 查看分支
git branch                 # 本地分支
git branch -r              # 远程分支
git branch -a              # 所有分支

# 创建分支
git branch feat/new-feature

# 切换分支
git checkout feat/new-feature

# 创建并切换分支
git checkout -b feat/new-feature

# 删除分支
git branch -d feat/new-feature        # 已合并
git branch -D feat/new-feature        # 强制删除

# 删除远程分支
git push origin --delete feat/new-feature
```

### 6.2 合并操作

```bash
# 合并分支
git checkout dev
git merge feat/new-feature

# 变基（推荐用于功能分支）
git checkout feat/new-feature
git rebase dev

# 解决冲突后继续变基
git rebase --continue

# 放弃变基
git rebase --abort
```

### 6.3 储藏操作

```bash
# 储藏当前更改
git stash

# 查看储藏列表
git stash list

# 恢复最近一次储藏
git stash pop

# 恢复指定储藏
git stash apply stash@{1}

# 删除储藏
git stash drop stash@{1}

# 清空所有储藏
git stash clear
```

### 6.4 撤销操作

```bash
# 撤销工作区更改
git checkout -- <file>

# 撤销暂存区更改
git reset HEAD <file>

# 撤销上次提交（保留更改）
git reset --soft HEAD~1

# 撤销上次提交（丢弃更改）
git reset --hard HEAD~1

# 撤销指定提交（创建新提交）
git revert <commit-id>
```

## 七、分支管理检查清单

### 7.1 创建分支前检查

- [ ] 基于正确的分支创建（feat/fix 基于 dev，hotfix 基于 main）
- [ ] 分支名称符合命名规范
- [ ] 本地分支已更新到最新

### 7.2 提交代码前检查

- [ ] 代码可以正常运行
- [ ] 提交信息符合规范
- [ ] 只提交相关文件
- [ ] 无敏感信息泄露

### 7.3 创建 PR 前检查

- [ ] 分支已推送到远程
- [ ] PR 标题符合规范
- [ ] PR 描述完整
- [ ] 关联了相关 Issue
- [ ] CI 检查通过

### 7.4 合并前检查

- [ ] 代码审查通过
- [ ] 所有评论已解决
- [ ] CI 检查通过
- [ ] 无冲突需要解决

## 八、国内 Git 平台适配

### 8.1 平台对比

| 特性 | Gitee | Coding.net | 极狐 GitLab | CNB |
|------|-------|------------|-------------|-----|
| 国内访问 | 快 | 快 | 快 | 快 |
| 免费私有仓库 | 有 | 有 | 有 | 有 |
| CI/CD | Gitee Go | Coding CI | 内置 GitLab CI | 内置 |
| 代码审查 | PR | MR | MR | MR |
| 适合场景 | 开源/小团队 | 中大型团队 | 企业私有化 | 云原生 |

### 8.2 Gitee 配置

```bash
# 设置 Gitee 远程仓库
git remote add origin https://gitee.com/<org>/<repo>.git

# SSH 配置
# ~/.ssh/config
Host gitee.com
    HostName gitee.com
    User git
    IdentityFile ~/.ssh/gitee_rsa
    PreferredAuthentications publickey
```

### 8.3 Coding.net 配置

```bash
# Coding 仓库地址格式
git remote add origin https://e.coding.net/<team>/<project>/<repo>.git

# SSH 地址
git remote add origin git@e.coding.net:<team>/<project>/<repo>.git
```

### 8.4 极狐 GitLab 配置

```bash
# 极狐 GitLab 地址格式
git remote add origin https://jihulab.com/<group>/<repo>.git

# 企业内部部署
git remote add origin https://gitlab.yourcompany.com/<group>/<repo>.git
```
