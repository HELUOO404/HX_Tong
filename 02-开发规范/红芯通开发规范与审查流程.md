## 一、代码规范

### 1.1 命名规范

#### 1.1.1 文件命名

| 类型 | 规范 | 示例 |
|------|------|------|
| 页面文件 | 小写+横线 | `room-edit.js`, `approval-detail.wxml` |
| 组件文件 | 小写+横线 | `meeting-card.js`, `time-picker.wxml` |
| 云函数目录 | 小写+横线 | `meetingroomService`, `adminService` |
| 工具类文件 | 小写+横线 | `errorHandler.js`, `validator.js` |
| 样式文件 | 与页面同名 | `room-edit.wxss` |

#### 1.1.2 变量命名

| 类型 | 规范 | 示例 |
|------|------|------|
| 普通变量 | 小写驼峰 | `userName`, `roomId`, `isLoading` |
| 常量 | 全大写+下划线 | `MAX_BOOKING_HOURS`, `DEFAULT_SIGN_IN_DEADLINE` |
| 布尔变量 | is/has/can 前缀 | `isAvailable`, `hasPermission`, `canApprove` |
| 数组变量 | 复数名词 | `users`, `bookings`, `timeSlots` |
| 对象变量 | 名词或形容词 | `roomInfo`, `formData`, `userInfo` |

#### 1.1.3 函数命名

| 类型 | 规范 | 示例 |
|------|------|------|
| 事件处理函数 | on+动作 | `onSubmit`, `onCancel`, `onSelectDate` |
| 数据加载函数 | load/get+名词 | `loadUsers`, `getRoomDetail` |
| 业务处理函数 | 动词短语 | `createBooking`, `approveBooking` |
| 工具函数 | 小写+下划线 | `format_date`, `calculate_score` |

#### 1.1.4 数据库字段命名

| 类型 | 规范 | 示例 |
|------|------|------|
| 集合名称 | 小写复数 | `users`, `rooms`, `bookings` |
| 字段名称 | 小写驼峰或下划线 | `userName` 或 `user_name` |
| 时间字段 | `xxxAt` 或 `xxx_time` | `createAt`, `updateTime` |
| 状态字段 | 小写 | `status`, `available` |

```javascript
 * @param {string} params.date - 预约日期
```

## 二、TDD 测试驱动开发

### 2.1 核心原则

**铁律**: 没有失败的测试，就不写生产代码

| 阶段 | 说明 |
|------|------|
| 🔴 红灯 | 先写测试，看它失败 |
| 🟢 绿灯 | 写最少的代码让它通过 |
| 🔵 重构 | 清理代码，保持测试通过 |

### 2.2 红-绿-重构流程

```
      ┌─────────────────────────────────────────┐
      │                                         │
      ▼                                         │
┌──────────┐    是    ┌──────────┐    是    ┌──────────┐
│   红灯    │ ──────▶ │   绿灯    │ ──────▶ │   重构    │
│ 写失败测试 │         │ 最少代码  │         │  清理代码  │
└──────────┘         └──────────┘         └──────────┘
      ▲                                        │
      │                                        │
      │         保持绿灯                        │
      └────────────────────────────────────────┘
```

### 2.3 开发流程

#### 步骤1: 红灯 - 编写失败的测试

```javascript
// 为新功能编写测试
test('会议室预约应扣除正确信誉分', async () => {
  // 场景：会议开始前5小时取消
  const cancelTime = new Date('2026-04-25 09:00'); // 会议时间
  const userCancelTime = new Date('2026-04-25 04:00'); // 取消时间

  const deduct = calculateCreditDeduct(cancelTime, userCancelTime);

  expect(deduct).toBe(-3); // 3-6小时前取消，扣3分
});
```

#### 步骤2: 验证红灯 - 确认测试失败

```bash
# 运行测试，确认失败原因正确
npm test path/to/test.js

# 预期：测试失败，显示 expected -3, got undefined
```

**验证要点**:
- 测试失败（不是报错）
- 失败信息符合预期
- 失败原因是功能缺失

#### 步骤3: 绿灯 - 编写最少代码

```javascript
// 写最少的代码让测试通过
function calculateCreditDeduct(startTime, cancelTime) {
  const diffHours = (startTime - cancelTime) / (1000 * 60 * 60);

  if (diffHours >= 6) return 0;      // ≥6小时前
  if (diffHours >= 3) return -3;      // 3-6小时前
  if (diffHours >= 1) return -5;      // 1-3小时前
  return -10;                         // <1小时
}
```

#### 步骤4: 验证绿灯 - 确认测试通过

```bash
npm test path/to/test.js

# 预期：所有测试通过
```

#### 步骤5: 重构 - 清理代码

- 消除重复代码
- 改善命名
- 提取辅助函数
- 保持测试绿灯

### 2.4 TDD实施要求

| 要求 | 说明 |
|------|------|
| 先写测试 | 功能代码编写前必须先编写测试 |
| 验证失败 | 必须执行测试确认失败原因正确 |
| 最少代码 | 只写让测试通过的最小代码 |
| 测试通过 | 必须确认所有测试通过 |
| 无例外 | 除非获得明确许可，否则不得跳过TDD |

### 2.5 何时使用TDD

| 场景 | 是否使用TDD |
|------|------------|
| 新功能开发 | ✅ 必须 |
| Bug修复 | ✅ 必须 |
| 重构 | ✅ 必须 |
| 行为变更 | ✅ 必须 |
| 一次性原型 | ❌ 可跳过 |
| 配置文件修改 | ❌ 可跳过 |

### 2.6 危险信号 - 立即停止

以下情况意味着违反了TDD原则，需要从头开始：

- [ ] 先写了代码再写测试
- [ ] 实现完了才补测试
- [ ] 测试立即通过
- [ ] 无法解释测试为什么失败
- [ ] "之后再补"测试
- [ ] "我已经手动测试过了"
- [ ] "后补测试也能达到相同目的"

**以上所有情况都意味着：删除代码。用 TDD 从头开始。**

### 2.7 TDD验证清单

在标记工作完成之前：

- [ ] 每个新函数/方法都有测试
- [ ] 在实现之前看到每个测试失败
- [ ] 每个测试因预期原因失败（功能缺失，不是拼写错误）
- [ ] 为每个测试编写了最少代码使其通过
- [ ] 所有测试通过
- [ ] 输出干净（没有错误、警告）
- [ ] 覆盖了边界情况和错误场景

## 四、代码文档管理规范

### 4.1 文档结构

每个大代码模块必须包含以下文档：

| 文档 | 位置 | 内容 |
|------|------|------|
| API文档 | `cloudfunctions/[模块名]/API_DOCUMENTATION.md` | 云函数接口文档 |
| 页面文档 | `docs/superpowers/specs/[模块名]/spec.md` | 页面规格说明 |
| 任务文档 | `docs/superpowers/specs/[模块名]/tasks.md` | 开发任务清单 |
| 检查文档 | `docs/superpowers/specs/[模块名]/checklist.md` | 验收检查清单 |

### 4.2 API文档格式

```markdown
# [模块名] API 文档

**最后更新**: YYYY-MM-DD
**维护人**: [姓名]
**版本**: x.x.x

## 接口列表

| action | 功能 | 参数 | 返回 | 状态 |
|--------|------|------|------|------|
| xxx_create | 创建 | {...} | {data} | ✅ |
| xxx_update | 更新 | {id, ...} | {success} | ⚠️ |
| xxx_delete | 删除 | {id} | {success} | ✅ |

## 更新日志

| 日期 | 版本 | 更新内容 | 更新人 |
|------|------|---------|--------|
| YYYY-MM-DD | x.x.x | 初始版本 | xxx |
| YYYY-MM-DD | x.x.x | 添加xxx接口 | xxx |
```

### 4.3 文档更新规则

| 规则 | 说明 |
|------|------|
| 即时更新 | 修改云函数或添加新接口后，立即更新API文档 |
| 每月检查 | 每月进行一次全量检查，确保文档不过时 |
| 版本同步 | 文档版本与代码版本保持一致 |
| 状态标记 | ✅ 正常 / ⚠️ 待更新 / ❌ 已废弃 |

### 4.4 文档状态检查

| 检查项 | 频率 | 负责人 |
|--------|------|--------|
| API文档完整性 | 每次接口变更 | 开发者 |
| 文档时效性 | 每周 | 项目经理 |
| 全量检查 | 每月 | 技术负责人 |

## 五、Git 工作流程

### 5.1 分支命名规范

| 分支类型 | 命名规范 | 示例 |
|---------|---------|------|
| 主分支 | master | master |
| 开发分支 | develop | develop |
| 功能分支 | feature/功能名 | feature/room-management |
| 修复分支 | fix/问题描述 | fix/booking-conflict |
| 发布分支 | release/版本号 | release/v1.0.0 |

### 5.2 提交信息规范

#### 5.2.1 提交信息格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

#### 5.2.2 Type 类型

| Type | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复bug |
| docs | 文档变更 |
| style | 代码格式（不影响功能） |
| refactor | 重构（不是新功能也不是修复bug） |
| perf | 性能优化 |
| test | 测试相关 |
| chore | 构建/工具变更 |

#### 5.2.3 提交信息示例

```
feat(booking): 添加会议室预约冲突检测

检测同一会议室同一时段只能有一个预约

Closes #123
```

```
fix(user): 修复用户信誉分计算错误

当baseScore为0时，计算结果错误

Fixes #456
```

```
docs(readme): 更新项目说明文档

添加部署说明
```

### 5.3 工作流程

```
develop 分支
    │
    ├── feature/room-management 分支 (开发中)
    │       │
```

### 6.0 必须检查项（低级错误防范）

**⚠️ 这类错误导致整个功能完全不可用，必须在代码审查时逐项检查**

#### 6.0.1 app.json 页面注册检查

| 检查项 | 说明 | 错误后果 |
|--------|------|---------|
| 新页面是否已注册 | 创建新页面后必须在 app.json 中添加 | 页面无法跳转，按钮点击无反应 |
| 页面路径是否正确 | 路径必须与实际文件路径一致 | 跳转失败 |
| 页面顺序是否正确 | pages 数组中第一个为首页 | 首页错误 |

```json
// ✅ 正确示例
{
  "pages": [
    "pages/index/index",
    "pages/admin/index",
    "pages/admin/users",
    "pages/admin/user-edit"
  ]
}

// ❌ 错误示例 - 缺少 user-edit 注册
{
  "pages": [
    "pages/index/index",
    "pages/admin/index",
    "pages/admin/users"
    // user-edit 缺失！
  ]
}
```

**审查时必须执行**:
1. 对比 `pages/` 目录下所有 `.js` 文件
2. 检查每个文件是否都在 `app.json` 的 pages 数组中
3. 确认每个页面的路径与实际文件路径完全一致

#### 6.0.2 API 返回数据结构检查

| 检查项 | 说明 | 错误后果 |
|--------|------|---------|
| 直接调用 vs ApiService | 两种方式返回结构不同 | 数据访问错误，页面闪退 |
| result vs result.data | wx.cloud.callFunction 返回 `{result: {...}}` | undefined 错误 |

```javascript
// ❌ 错误 - 直接调用云函数但数据结构理解错误
const { result } = await wx.cloud.callFunction({
  name: 'adminService',
  data: { action: 'xxx', params: {} }
})
if (result.data.list) {  // ❌ result 是 {code, data, message}，不是 result.data
  // ...
}

// ✅ 正确 - 使用 ApiService（推荐）
const api = require('../../逻辑层/services/api.service').getInstance()
const res = await api.admin.user.getList({ keyword, pageSize })
if (res.list) {  // ✅ res 已经是处理后的数据
  // ...
}

// ✅ 正确 - 直接调用云函数
const { result } = await wx.cloud.callFunction({
  name: 'adminService',
  data: { action: 'xxx', params: {} }
})
if (result.code === 200 && result.data.list) {  // ✅ 正确访问
  // ...
}
```

**审查时必须执行**:
1. 检查所有 API 调用是否统一使用 ApiService
2. 如果直接使用 `wx.cloud.callFunction`，必须验证返回结构
3. 添加注释说明返回数据的结构

#### 6.0.3 权限字段一致性检查

| 检查项 | 说明 | 错误后果 |
|--------|------|---------|
| level vs roleLevel | 登录返回与前端检查的字段必须一致 | 权限验证永远失败 |
| 角色判断逻辑 | `role` vs `level` 必须区分清楚 | 非管理员可访问管理页面 |

```javascript
// ❌ 错误 - 字段名不一致
// 登录返回: { roleLevel: 0, role: 'superAdmin' }
// 前端检查: if (adminInfo.level !== 0)  // ❌ level 是 undefined

// ✅ 正确 - 统一字段名
// 登录返回: { level: 0, roleLevel: 0, role: 'superAdmin' }
// 前端检查: if (adminInfo.level !== 0)

// ✅ 正确 - 兼容处理
const adminLevel = adminInfo.level !== undefined ? adminInfo.level : 
                   (adminInfo.roleLevel !== undefined ? adminInfo.roleLevel : 99)
if (adminLevel !== 0) {
  // 无权限
}
```

**审查时必须执行**:
1. 检查云函数登录返回的数据结构
2. 检查前端权限检查使用的字段名
3. 确保两者完全一致，或添加兼容处理
