## 一、架构概述

本项目采用**四层架构设计**，实现关注点分离，提高代码可维护性和可扩展性。

### 1.1 架构分层

```
┌─────────────────────────────────────────┐
│              皮肤层 (Skin)               │
│         主题系统、样式变量、动态换肤        │
├─────────────────────────────────────────┤
│             结构层 (Structure)           │
│         页面结构、组件规范、布局系统        │
├─────────────────────────────────────────┤
│              逻辑层 (Logic)              │
│         业务逻辑、API服务、数据处理        │
├─────────────────────────────────────────┤
│            数据互通层 (Data)             │
│         全局配置、状态管理、常量定义        │
├─────────────────────────────────────────┤
│           微信云开发 (Cloud Base)         │
│         云函数、云数据库、云存储           │
└─────────────────────────────────────────┘
```

### 1.2 层间依赖关系

```
皮肤层 ← 结构层 ← 逻辑层 ← 数据互通层 ← 云开发
```

**依赖规则**：
- 上层可以依赖下层
- 下层禁止依赖上层
- 同层之间可以相互依赖

## 二、各层职责详解

### 2.1 数据互通层

**目录**：`miniprogram/数据互通层/`

**职责**：
- 存储全局配置（信誉分规则、预约规则、用户信息完善规则等）
- 管理全局状态（用户信息、主题状态等）
- 定义常量（错误码、状态枚举等）

**规范**：
- 配置项必须集中管理，禁止散落在业务代码中
- 状态管理使用统一的 Store 模式
- 常量使用大写字母 + 下划线命名

**核心文件**：
| 文件路径 | 说明 |
|---------|------|
| `数据互通层/config/global.config.js` | 全局业务配置（信誉分规则、预约规则等） |
| `数据互通层/config/constants.js` | 常量定义（错误码、状态枚举等） |

### 2.2 逻辑层

**目录**：`miniprogram/逻辑层/`

**职责**：
- 封装业务逻辑
- 统一 API 调用
- 数据处理和转换

**规范**：
- 服务类使用单例模式
- API 调用统一封装，禁止直接调用 `wx.cloud`
- 业务逻辑与页面逻辑分离

**核心文件**：
| 文件路径 | 说明 |
|---------|------|
| `逻辑层/services/api.service.js` | 统一 API 调用服务 |

### 2.3 结构层

**目录**：`miniprogram/结构层/`（实际对应 `miniprogram/pages/`）

**职责**：
- 定义页面结构
- 封装可复用组件
- 规范布局系统

**规范**：
- 页面文件使用标准结构（js/json/wxml/wxss）
- 组件必须包含完整的生命周期
- 布局使用 flex，禁止使用 float

### 2.4 皮肤层

**目录**：`miniprogram/皮肤层/`（实际对应 `miniprogram/theme/`）

**职责**：
- 管理主题配置
- 定义样式变量
- 实现动态换肤

**规范**：
- 颜色必须使用 CSS 变量
- 主题配置使用 JSON 格式
- 支持运行时切换主题

**核心文件**：
| 文件路径 | 说明 |
|---------|------|
| `theme/theme-config.js` | 主题配置数据 |
| `theme/theme-utils.js` | 主题工具函数 |
| `theme/theme-mixin.js` | 页面主题混入 |

## 三、云函数架构

### 3.1 服务大合集模式

采用**服务大合集**架构，按业务领域划分云函数：

```
cloudfunctions/
├── meetingroomService/      # 会议室服务（13 个 API）
│   ├── handlers/
│   │   ├── booking/         # 预约相关
│   │   ├── room/            # 会议室查询
│   │   └── manage/          # 管理相关
│   ├── utils/
│   │   ├── permission.js    # 权限工具
│   │   ├── response.js      # 响应封装
│   │   └── validator.js     # 参数校验
│   ├── index.js             # 入口文件
│   └── package.json
├── userService/             # 用户服务（6 个 API）
├── adminService/            # 管理员服务（4 个 API）
└── systemService/           # 系统服务（2 个 API）
```

### 3.2 云函数命名规范

**统一格式**：`{业务领域}Service`

| 云函数 | 说明 | API 数量 |
|--------|------|----------|
| `meetingroomService` | 会议室服务 | 13 |
| `userService` | 用户服务 | 6 |
| `adminService` | 管理员服务 | 4 |
| `systemService` | 系统服务 | 2 |

### 3.3 云函数入口规范

```javascript
/**
 * @file meetingroomService 云函数主入口
 * @description 会议室服务大合集
 */

const cloud = require('wx-server-sdk')
cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })

// 处理器映射
const handlers = {
  'meetingroom_booking_create': require('./handlers/booking/create'),
  'meetingroom_booking_cancel': require('./handlers/booking/cancel'),
  // ...
}

exports.main = async (event, context) => {
  const { action, params = {} } = event
  
  if (!action) {
    return { code: 400, message: '缺少 action 参数' }
  }
  
  const handler = handlers[action]
  if (!handler) {
    return { code: 404, message: `未知的 action: ${action}` }
  }
  
  try {
    return await handler(params, cloud)
  } catch (error) {
    console.error(`[meetingroomService] ${action} 执行失败:`, error)
    return { code: 500, message: '服务器内部错误' }
  }
}
```

## 四、状态管理架构

### 4.1 UserStore 全局状态管理

采用**单例模式**实现全局用户状态管理：

```
┌─────────────────────────────────────────┐
│              UserStore                  │
│  ┌─────────────┐     ┌─────────────┐   │
│  │  _userInfo  │     │  _isLogin   │   │
│  │  (用户信息)  │     │  (登录状态)  │   │
│  └─────────────┘     └─────────────┘   │
│           │                              │
│           ▼                              │
│  ┌─────────────────────────────────┐   │
│  │        _listeners[]             │   │
│  │      (状态变化监听器)            │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**核心原则**：
- 全局唯一的用户信息存储中心
- 所有模块统一从 UserStore 调取用户信息
- 禁止在各模块中分散存储全局信息

### 4.2 使用示例

```javascript
// 页面中订阅用户信息
const UserStore = require('../../stores/userStore')

Page({
  _unsubscribeUserStore: null,

  onLoad() {
    const userStore = UserStore.getInstance()
    this._unsubscribeUserStore = userStore.subscribe((state) => {
      this.setData({
        userInfo: state.userInfo,
        isLogin: state.isLogin
      })
    })
  },

  onUnload() {
    if (this._unsubscribeUserStore) {
      this._unsubscribeUserStore()
    }
  }
})
```

## 五、主题系统架构

### 5.1 技术限制说明

微信小程序存在以下技术限制：
- 不支持 `:root` 伪类定义 CSS 变量
- 不支持动态加载外部 WXSS
- 不支持动态修改全局样式
- 无法运行时下载并保存样式文件

### 5.2 解决方案

采用**预置主题 + 动态切换**方案：

```
┌─────────────────────────────────────────┐
│              主题配置层                  │
│  ┌──────────────┐  ┌──────────────┐    │
│  │theme-config.js│  │theme-utils.js│    │
│  │  主题配置数据  │  │  主题工具函数  │    │
│  └──────────────┘  └──────────────┘    │
└─────────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│              应用层                      │
│  ┌──────────────┐  ┌──────────────┐    │
│  │   页面 JS     │  │   WXML        │    │
│  │ 引入theme-mixin│  │ 内联样式绑定  │    │
│  └──────────────┘  └──────────────┘    │
└─────────────────────────────────────────┘
```

### 5.3 使用示例

```javascript
// 页面引入主题混入
const ThemeMixin = require('../../theme/theme-mixin')

Page({
  ...ThemeMixin,
  
  data: {
    // theme、themeStyle、currentThemeId 由混入自动注入
  }
})
```

```xml
<!-- WXML 中使用主题变量 -->
<view class="page" style="background: {{theme.background}};">
  <view class="header" style="background: {{theme.primary}};">
    <text style="color: {{theme.textPrimary}};">标题</text>
  </view>
</view>
```

## 六、权限架构

### 6.1 权限金字塔

采用**数字等级**体系，数字越小权限越高：

| 等级 | 角色 | 权限范围 |
|------|------|----------|
| 0 | 超级管理员 (superAdmin) | 所有功能，包括创建权限标签、修改用户角色 |
| 5 | 管理员 (admin) | 会议室管理、预约审批、用户管理（除角色修改） |
| 10 | 普通用户 (user) | 预约会议室、查看个人预约 |

### 6.2 权限检查工具

```javascript
// utils/permission.js
const ROLE_LEVELS = {
  superAdmin: 0,
  admin: 5,
  user: 10
}

// 检查是否有权限（等级数字越小权限越高）
function hasPermission(userRole, requiredLevel) {
  const userLevel = ROLE_LEVELS[userRole] || 99
  return userLevel <= requiredLevel
}

// 检查是否为管理员
function isAdmin(userRole) {
  return hasPermission(userRole, ROLE_LEVELS.admin)
}
```

## 八、架构检查清单

### 8.1 代码审查检查项

- [ ] 遵循四层架构，层间依赖正确
- [ ] 业务逻辑封装在服务层，不在页面中直接处理
- [ ] 使用 UserStore 管理全局用户信息
- [ ] 页面正确引入 theme-mixin 支持主题切换
- [ ] 云函数使用处理器映射模式
- [ ] 权限检查使用统一的 permission.js 工具

### 8.2 禁止行为

- [ ] 禁止在页面中直接调用 `wx.cloud`
- [ ] 禁止在各模块中存储用户信息副本
- [ ] 禁止下层依赖上层（如服务层依赖页面）
- [ ] 禁止在云函数中直接操作数据库（使用封装层）
- [ ] 禁止跨层调用（如页面直接查询数据库）
