## 一、项目概述

### 1.1 项目背景与意义

#### 1.1.1 学院学生服务现状分析

深圳职业技术大学集成电路学院现有在校学生约1000人，目前学生日常服务主要依赖以下渠道：

- **教务处网站**：查询课程表、考试成绩，但界面不够友好，移动端体验差
- **学院官网**：查看通知公告，但更新不及时，信息分散
- **微信群**：班级群、年级群信息繁杂，重要通知容易被淹没
- **纸质流程**：会议室预约、失物招领等需要线下办理，效率低下
- **多个系统**：体测成绩、课程信息分散在不同系统，学生需要反复切换
- **第三方小程序**：如"学深星球"等，但广告较多，数据安全存疑

这种分散化的服务模式存在以下问题：
1. **信息获取不便**：学生需要在多个平台之间切换，浪费时间
2. **通知传达效率低**：重要信息容易被群消息淹没，学生错过关键通知
3. **资源预约困难**：会议室等资源预约流程繁琐，缺乏统一管理平台
4. **缺乏互动渠道**：学生之间缺乏便捷的交流互动平台
5. **数据孤岛现象**：各系统数据不互通，无法形成统一的学生服务视图
6. **第三方平台问题**：现有小程序广告多、数据不透明、难以定制化

#### 1.1.2 红芯通小程序开发必要性

基于上述现状，开发一款学院自主掌控的集成化生活服务小程序具有以下必要性：

**（1）提升学生服务体验**
- 一站式解决学生日常需求，无需在多个系统间切换
- 移动端优先设计，随时随地获取服务
- 零广告干扰，纯净使用体验
- 现代化UI设计，视觉体验更佳

**（2）保障数据安全可控**
- 学院自主管理，数据不出校园
- 符合《个人信息保护法》《数据安全法》要求
- 无第三方数据收集和商业化使用风险
- 分阶段身份认证，最终与学校系统打通

**（3）提高学院管理效率**
- 信息发布直达学生，提高传达效率
- 资源预约线上化，减少人工管理成本
- 会议室预约自动化，冲突检测智能化
- 数据统计分析，支持管理决策

**（4）推进学院数字化建设**
- 构建学院数字化服务体系
- 积累学生服务数据，支持数据驱动决策
- 提升学院信息化水平，打造智慧校园示范
- 与学校统一身份认证对接，规范身份管理

**（5）响应学校信息化建设要求**
- 符合学校"十四五"信息化建设规划
- 落实"以学生为中心"的服务理念
- 探索新技术在教育教学中的应用

#### 1.1.3 项目预期目标

**总体目标**：开发并上线"红芯通"小程序，为集成电路学院师生提供一站式生活服务，打造学院数字化服务品牌。

- **痛点认知**：对现有第三方小程序广告多、体验差有切身感受

基于前期调研（详见附录A：需求调研报告），用户核心需求排序如下：

**第二优先级（V1.1版本重要功能）**：

9. **学习资源**（需求度：55%）：分享学习资料和笔记

## 二、技术方案

### 2.1 开发架构选择

#### 2.1.1 Trae Solo模式开发优势

本项目采用**Trae Solo模式**进行开发，这是一种基于AI的代码生成开发模式，具有以下显著优势：

**（1）开发效率大幅提升**
- 通过自然语言描述即可生成代码，无需从零编写
- 代码生成效率比传统开发提升70%以上
- 快速原型验证，缩短开发周期

**（2）降低技术门槛**
- 零基础开发者也能快速上手
- AI辅助代码审查，减少错误
- 自动生成注释和文档

**（3）代码质量保障**
- AI生成的代码符合最佳实践
- 自动遵循微信小程序开发规范
- 内置代码优化建议

**（4）成本优势明显**
- 无需雇佣专业开发团队
- 学生团队即可完成开发
- 降低项目整体成本

**Trae Solo模式开发流程**：

```
需求描述（自然语言）
    ↓
Trae AI代码生成
    ↓
代码审查与优化
    ↓
功能测试
    ↓
迭代完善
```

**核心技术组件**：

#### 2.1.3 身份验证方案

本项目采用**分阶段身份验证策略**，平衡快速上线与规范管理的双重需求：

**阶段一：MVP版本（微信登录）**
- **实现方式**：微信小程序一键登录
- **优势**：开发快速，用户体验好，无需额外审批
- **数据存储**：微信openid + 用户基本信息（昵称、头像）
- **适用场景**：快速上线，验证产品价值

**阶段二：过渡期（微信登录+学校账号绑定）**
- **实现方式**：已微信登录用户需绑定学校统一身份认证账号
- **绑定流程**：
  1. 用户在个人中心点击"绑定学校账号"
  2. 跳转学校统一认证页面
  3. 认证成功后，关联微信openid与学校学号
  4. 后续可使用任一方式登录
- **数据迁移**：确保用户数据（预约记录等）不丢失

**阶段三：V2.0版本（强制学校统一认证）**
- **实现方式**：必须完成学校统一身份认证才能使用
- **前置条件**：
  - 学院领导审批同意接入
  - 与学校网络中心对接完成
  - 获得统一身份认证接口权限
- **优势**：身份真实可靠，支持教务数据对接，符合学校规范

**身份验证技术实现**：

| 验证阶段 | 技术方案 | 数据标识 | 安全级别 |
|---------|---------|---------|---------|
| MVP版本 | 微信登录 | openid | 中 |
| 过渡期 | 微信登录+学校账号绑定 | openid+学号 | 高 |
| V2.0版本 | 学校统一身份认证 | 学号 | 最高 |

```javascript
// utils/theme.js
const themeManager = {
  // 当前主题
  currentTheme: 'default',
  
  // 切换主题
  switchTheme(themeName) {
    const theme = themes[themeName];
    if (!theme) return;
    
    // 生成 CSS 字符串
    const cssVars = this.generateCSSVars(theme);
    
    // 注入到页面
    this.injectCSS(cssVars);
    
    // 保存用户偏好
    wx.setStorageSync('userTheme', themeName);
    this.currentTheme = themeName;
  },
  
  // 生成 CSS 变量
  generateCSSVars(theme) {
    return `
      --primary-color: ${theme.colors.primary};
      --primary-light: ${theme.colors.primaryLight};
      --background-color: ${theme.colors.background};
      /* ... 其他变量 */
    `;
  },
  
  // 初始化主题
  init() {
    const savedTheme = wx.getStorageSync('userTheme') || 'default';
    this.switchTheme(savedTheme);
  }
};
```

**模块四：信誉分机制模块（V1.3版本）**

*4.1 基础信誉分体系*
- **初始分值**：所有学生初始默认 100 分，作为使用书院/会议室资源的信用凭证
- **分值范围**：0-150 分，上限封顶防止恶意刷分
- **数据存储**：云数据库存储，支持历史记录追溯

*4.2 加分规则*
| 加分场景 | 加分分值 | 限制条件 |
|---------|---------|---------|
| 按时履约 | +2分/次 | 每月上限10分 |
| 无违规使用（连续30天） | +5分 | 每月评定一次 |
| 主动归还设备并确认完好 | +3分/次 | 需管理员确认 |
| 好评反馈（使用评价） | +1分/次 | 每月上限5分 |
| 举报违规行为（核实） | +5分/次 | 无上限 |
| 特殊贡献（活动组织等） | +10-20分 | 管理员评定 |

*4.3 扣分规则*
| 扣分场景 | 扣分分值 | 备注 |
|---------|---------|------|
| 预约未到（未取消） | -10分/次 | 提前2小时可免责取消 |
| 超时未离场（15分钟内） | -5分/次 | 系统提醒后及时离场可减半 |
| 超时未离场（15分钟以上） | -15分/次 | 影响后续预约 |
| 违规使用场地 | -20分/次 | 如吸烟、饮食、噪音等 |
| 损坏物品（轻微） | -15分/次 | 需照价赔偿 |
| 损坏物品（严重） | -30分/次 | 暂停预约权限30天 |
| 恶意申请（虚假用途） | -25分/次 | 暂停预约权限15天 |
| 借用设备未归还 | -20分/次 | 每日追加-5分直至归还 |

*4.4 等级体系*
| 等级 | 分数区间 | 权益说明 | 限制措施 |
|-----|---------|---------|---------|
| **优秀** | 120-150分 | 优先预约权、延时申请免审核、专属标识 | 无 |
| **良好** | 90-119分 | 正常预约权限、延时申请需审核 | 无 |
| **合格** | 60-89分 | 正常预约权限、延时申请受限 | 每日限预约1次 |
| **受限** | 0-59分 | 暂停会议室预约权限、禁止延时申请 | 需完成信用修复任务 |

*4.5 功能支撑*
- **信誉分查询**：实时显示当前分数、等级、排名（前10%显示）
- **奖惩记录查看**：详细记录每笔分数变动的时间、事由、操作人
- **等级公示**：个人中心展示等级徽章，优秀用户特殊标识
- **违规申诉**：对扣分有异议可提交申诉，管理员3个工作日内处理
- **信用修复**：受限用户可通过完成指定任务（如志愿服务）恢复分数

*4.6 联动规则*
- 信誉分低于 60 分：自动暂停会议室预约权限
- 信誉分低于 80 分：限制延时申请权限（仅可申请1小时以内）
- 信誉分 120 分以上：延时申请免审核，直接通过
- 信誉分排名 Top 10：学期末公示表彰

**模块五：书院延时使用申请模块（V1.3版本）**

*5.1 申请入口*
- **首页快捷入口**：首页显眼位置设置"延时申请"快捷按钮
- **书院服务专区**：集中展示书院相关服务，延时申请为首要入口
- **预约记录联动**：已有预约可一键申请延时

*5.2 申请信息填写*
| 字段 | 必填 | 说明 |
|-----|------|------|
| 延时时长 | 是 | 选项：30分钟/1小时/2小时/3小时/自定义 |
| 使用用途 | 是 | 文本描述，不少于10字 |
| 使用区域 | 是 | 选择具体书院/会议室 |
| 联系方式 | 是 | 手机号，确保紧急联系 |
| 原预约单号 | 否 | 如为已有预约延时，自动关联 |
| 备注说明 | 否 | 特殊情况说明 |

*5.3 智能校验机制*
- **信誉分校验**：低于80分限制申请时长（最多1小时）
- **低于60分**：直接禁止提交申请，提示"信用不足"
- **时间冲突校验**：自动检测申请时段是否与其他预约冲突
- **频次限制**：同一区域每日最多申请1次延时
- **历史违规校验**：近期有违规记录需额外说明

*5.4 审核流程*
```
学生提交申请
    ↓
系统自动校验（信誉分、冲突检测）
    ↓
校验通过 → 推送管理员审核
    ↓
管理员操作：通过 / 驳回 / 转交 / 备注
    ↓
系统通知学生审核结果
```

*5.5 状态通知*
- **提交成功通知**："您的延时申请已提交，预计30分钟内审核"
- **审核通过通知**："延时申请已通过，可使用至XX时间"
- **审核驳回通知**："延时申请未通过，原因：XX"
- **延时到期提醒**：到期前10分钟推送提醒"您的延时即将到期，请做好离场准备"
- **即将超时预警**：超时前5分钟再次提醒

*5.6 记录管理*
- **学生端**：
  - 查看历史申请记录（时间、区域、时长、状态）
  - 查看当前进行中的延时申请
  - 取消待审核的申请（审核前可取消）
  
- **管理端**：
  - 查看所有待审核申请列表
  - 按区域/时间筛选申请记录
  - 申请统计报表（按日/周/月）
  - 导出申请记录Excel

*5.7 特殊场景处理*
- **紧急情况**：设置"紧急延时"通道，信誉分要求降低，但需事后补交说明
- **批量延时**：活动组织者可申请批量延时（需提前24小时）
- **自动通过**：信誉分120分以上且时长≤1小时的申请自动通过

### 2.3 数据安全方案

#### 2.3.1 用户数据保护措施

**数据分类与保护级别**：

| 数据类型 | 保护级别 | 保护措施 |
|---------|---------|---------|
| 用户身份信息（学号、姓名） | 高 | 加密存储，访问控制 |
| 微信openid | 中 | 加密存储，仅用于登录 |
| 预约记录 | 中 | 用户只能查看自己的记录 |
| 信誉分记录 | 高 | 加密存储，防篡改，操作留痕 |
| 延时申请记录 | 中 | 用户只能查看自己的申请 |
| 课程信息 | 低 | 公开信息，无需特殊保护 |
| 系统日志 | 低 | 定期清理，匿名化处理 |

**技术保护措施**：

**（1）数据加密**
- 传输加密：所有数据传输使用HTTPS协议
- 存储加密：敏感数据使用AES加密存储
- 密钥管理：密钥分离存储，定期更换

**（2）访问控制**
- 身份认证：微信登录+Token验证
- 权限控制：基于角色的访问控制（RBAC）
- 数据隔离：用户只能访问自己的数据

**（3）安全审计**
- 操作日志：记录用户关键操作
- 异常监控：实时监控异常访问行为
- 定期审计：定期审查数据访问日志

#### 2.3.2 系统权限管理设计

**角色体系**：

**（1）学生用户（默认角色）**
- 权限范围：
  - 查看个人课程表
  - 预约会议室
  - 查看我的预约记录
  - 管理个人账户信息
  - 查看个人信誉分及记录
  - 提交延时申请（需满足信誉分要求）
  - 查看我的延时申请记录
- 限制：不能审批预约，不能查看他人预约，不能修改信誉分

**（2）管理员（教师/学生干部）**
- 权限范围：
  - 学生用户的所有权限
  - 审批会议室预约申请
  - 管理会议室资源信息
  - 查看预约统计数据
  - 审批延时申请
  - 管理用户信誉分（加分/扣分操作）
  - 查看信誉分统计报表
  - 处理信誉分申诉
- 限制：不能查看学生个人隐私数据（除信誉分相关）

**（3）超级管理员（系统管理员）**
- 权限范围：
  - 所有管理员的权限
  - 用户管理（禁用/启用账号）
  - 系统配置（功能开关、参数设置）
  - 数据管理（备份、恢复）

#### 2.3.3 数据备份与恢复机制

**备份策略**：

**（1）自动备份**
- 备份频率：每天凌晨2点自动备份
- 备份内容：云数据库全量数据
- 备份保留：保留最近30天备份
- 存储位置：云存储多地域冗余存储

**（2）手动备份**
- 触发条件：重大更新前、数据迁移前
- 操作方式：管理员手动触发备份

**恢复机制**：
- 恢复时间目标（RTO）：4小时内恢复服务
- 恢复点目标（RPO）：最多丢失1天数据

#### 3.1.1 需求确认与架构设计阶段（第1-2周）

**阶段目标**：完成需求调研，确定技术方案，搭建开发环境

**主要任务**：
- 完成学生需求调研（问卷设计、发放、统计分析）
- 完成现有服务痛点分析（重点分析学深星球等竞品的不足）
- 完成竞品分析报告（与学深星球等对比）
- 确定技术架构方案（身份验证分阶段策略）
- 搭建开发环境（Trae、微信开发者工具、云开发）
- 完成技术架构设计文档

**阶段交付物**：
- 《需求调研报告》

**里程碑**：需求确认完成，技术方案确定

#### 3.1.2 MVP版本开发与测试阶段（第3-6周）

**阶段目标**：完成MVP版本3大核心功能开发，通过功能测试

**主要任务**：
- 第3周：完成用户管理模块开发（微信登录+个人账户）
- 第4周：完成课程表模块开发（展示+导入）
- 第5周：完成会议室预约模块开发（预约+审批+冲突检测）
- 第6周：功能测试、bug修复、性能优化、试运行准备

**阶段交付物**：
- 3大核心功能模块代码
- 《功能测试报告》
- 《性能测试报告》
- 可运行的MVP测试版本

**里程碑**：MVP版本开发完成，通过内部测试

#### 3.1.3 MVP版本试运行与优化阶段（第7-8周）

**阶段目标**：小范围试运行，收集反馈，优化完善

**主要任务**：
- 选择50名种子用户进行试运行（学生会、班干部优先）

- 优化用户体验
- 准备正式上线

**阶段交付物**：
- 《试运行反馈报告》
- 优化后的MVP正式版本
- 用户帮助文档

**里程碑**：试运行通过，MVP版本正式上线

**V1.1版本（第9-12周）**：

**V1.2版本（第13-16周）**：

- 移动端优先，体验流畅

- 聚焦学生高频使用场景（会议室预约为刚需）

- 支持持续迭代优化
