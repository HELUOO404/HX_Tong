## 一、项目概述

### 1.1 项目背景

开发一款会议室预约小程序，为用户提供便捷的会议室资源预约服务，解决传统线下预约流程繁琐、效率低下的问题。

### 1.2 项目目标

开发并上线会议室预约小程序，为用户提供一站式资源预约服务。

**核心目标**：

| 目标类别 | 具体指标 | 完成时间 |
|---------|---------|---------|
| 功能覆盖 | 完成 3 大核心功能（用户管理 + 会议室预约 + 管理后台） | 第 6 周末 |
| 用户覆盖 | 首月用户覆盖率达 50% | 上线后 1 个月 |
| 用户满意度 | 用户满意度 ≥ 75% | 上线后 1 个月 |

### 1.3 目标用户群体

| 用户类型 | 规模 | 占比 |
|---------|------|------|
| 核心用户（学生/员工） | 约 1000 人 | 95% |
| 扩展用户（管理员） | 约 50 人 | 5% |

**用户特征**：

- **年龄分布**：18-45 岁
- **技术接受度**：高，熟练使用智能手机
- **使用习惯**：偏好移动端，习惯使用微信生态
- **需求特点**：注重便捷性、实时性

### 2.2 模块一：用户管理模块

#### 2.2.1 功能描述

| 功能点 | 优先级 | 功能说明 |
|-------|-------|---------|
| 微信一键登录 | P0 | 用户通过微信授权快速登录，获取 openid、昵称、头像 |
| 强制信息完善 | P0 | 首次登录需完善姓名、部门、手机号等必要信息 |

#### 2.2.2 用户场景与交互流程

```

    ↓

    ↓

    ↓
系统检测：是否已注册？
    ├── 已注册 → 进入首页
    └── 未注册 → 跳转信息完善页面

            ├─ 姓名
            ├─ 部门/班级
            └─ 手机号
                    ↓
            提交注册 → 创建用户记录
                    ↓
            进入首页
```

**场景二：查看个人信息**

```
用户点击 TabBar「我的」
    ↓
展示个人信息卡片：
├─ 头像、昵称
├─ 姓名、部门
└─ 身份标签
    ↓
点击「编辑资料」
    ↓
可修改：手机号、头像
    ↓
保存更新
```

### 2.3 模块二：会议室预约模块

#### 2.3.1 功能描述

| 功能点 | 优先级 | 功能说明 |
|-------|-------|---------|
| 会议室展示 | P0 | 展示会议室图片、容量、设备、位置等信息 |
| 时段选择 | P0 | 日历视图展示可预约时段，已预约时段置灰 |
| 在线预约 | P0 | 填写预约信息（日期、时段、用途、人数、联系方式） |
| 冲突检测 | P0 | 自动检测时间段冲突，防止重复预约 |
| 分场景审批 | P0 | 普通时段自动通过，特殊时段需管理员审批 |
| 预约管理 | P0 | 查看预约状态、取消预约 |
| 候补机制 | P1 | 已满时段可加入候补队列，有释放时自动提醒 |

#### 2.3.2 用户场景与交互流程

**场景一：预约会议室**

```
用户进入「预约」Tab
    ↓
浏览会议室列表（卡片展示）
    ↓
点击目标会议室
    ↓
查看会议室详情：
├─ 图片轮播
├─ 基本信息（容量、设备）
├─ 位置说明
└─ 预约规则
    ↓
点击「立即预约」
    ↓
选择预约日期（日历组件）
    ↓
选择时段（时间轴展示可预约/已预约状态）
    ↓
填写预约信息：
├─ 用途描述（必填）
├─ 参与人数
└─ 联系人电话
    ↓
提交预约
    ↓
系统检测：
    ├── 无冲突 + 普通时段 → 自动通过 → 预约成功
    └── 无冲突 + 特殊时段 → 提交审批 → 等待审核
```

**场景二：查看我的预约**

```
用户进入「我的」页面
    ↓
点击「我的预约」
    ↓
展示预约列表（按时间倒序）：
├─ 待审批（黄色标签）
├─ 已通过（绿色标签）
├─ 已拒绝（红色标签）
├─ 已取消（灰色标签）
└─ 已完成（蓝色标签）
    ↓
点击某条预约
    ↓
查看预约详情：
├─ 会议室信息
├─ 预约时段
├─ 用途说明
├─ 当前状态
└─ 操作按钮（取消预约）
```

**场景三：取消预约**

```
用户在预约详情页
    ↓
点击「取消预约」按钮
    ↓
确认弹窗：确认取消？
    ↓
确认后：
    ├── 待审批状态 → 直接取消
    └── 已通过状态 → 取消并释放时段
    ↓
更新预约状态为「已取消」
    ↓
如有候补用户，触发候补转正流程
```

#### 2.3.3 预约规则说明

| 规则项 | 说明 |
|-------|------|
| 预约时段 | 08:00 - 22:00（根据会议室配置） |
| 最小预约时长 | 1 小时 |
| 最大预约时长 | 4 小时 |
| 提前预约时间 | 至少提前 2 小时 |
| 取消免责时间 | 开始前 2 小时可免费取消 |
| 审批规则 | 工作日 18:00 后及周末时段需审批 |

## 三、非功能需求

### 3.1 性能需求

| 指标项 | 目标值 | 说明 |
|-------|-------|------|
| 页面首屏加载时间 | ≤ 2 秒 | 从点击到首屏渲染完成 |
| 云函数响应时间 | ≤ 1 秒 | API 接口平均响应时间 |
| 列表页滑动帧率 | ≥ 50 fps | 保证流畅的滚动体验 |
| 并发用户支持 | ≥ 100 人 | 同时在线预约操作 |
| 数据库查询响应 | ≤ 500 ms | 单次查询耗时 |

### 3.2 安全性需求

| 安全项 | 要求 |
|-------|------|
| 数据传输 | 所有数据传输使用 HTTPS 加密 |
| 身份认证 | 基于微信 openid + Token 验证 |
| 权限控制 | 基于角色的访问控制（RBAC） |
| 数据隔离 | 用户只能访问自己的数据 |
| 敏感信息 | 手机号等敏感信息加密存储 |
| 输入验证 | 所有输入参数在服务端验证 |
| 操作审计 | 关键操作记录系统日志 |

### 3.3 兼容性需求

| 兼容项 | 要求 |
|-------|------|
| 微信基础库 | 支持 2.2.3 及以上版本 |
| 设备系统 | 支持 iOS 10+ / Android 5.0+ |
| 屏幕适配 | 适配主流手机屏幕尺寸 |
| 微信版本 | 支持微信 7.0 及以上版本 |

### 3.4 可维护性需求

| 维护项 | 要求 |
|-------|------|
| 代码规范 | 遵循微信小程序开发规范 |
| 注释要求 | 关键逻辑必须添加注释 |
| 错误处理 | 统一错误码和错误提示 |
| 日志记录 | 关键操作和异常记录日志 |
| 数据备份 | 每日自动备份数据库 |

### 3.5 可用性需求

| 可用项 | 要求 |
|-------|------|
| 系统可用性 | ≥ 99.5%（月度） |
| 故障恢复时间 | ≤ 4 小时 |
| 数据恢复点目标 | 最多丢失 1 天数据 |

## 四、MVP 范围界定

| 模块 | 功能点 | 优先级 |

### 4.2 必要用户交互流程

MVP 版本必须支持以下核心业务流程：

```
【流程一：用户注册登录】
打开小程序 → 微信授权 → 信息完善 → 进入首页

【流程二：会议室预约】
选择会议室 → 选择日期 → 选择时段 → 填写信息 → 提交预约 → 获得结果

【流程三：预约审批】
管理员查看待审批 → 查看详情 → 通过/拒绝 → 用户收到通知

【流程四：预约管理】
用户查看我的预约 → 查看详情 → 取消预约（如需）
```

### 4.3 基础数据结构设计

#### 4.3.1 数据库集合清单

| 集合名 | 说明 | 主要用途 |
|-------|------|---------|
| users | 用户表 | 存储用户信息和身份标签 |
| rooms | 会议室表 | 存储会议室资源信息 |
| bookings | 预约记录表 | 存储所有预约申请 |
| booking_rules | 预约规则表 | 配置特殊时段审批规则 |

#### 4.3.2 核心数据结构

**users 集合**：

```javascript
{
  _id: String,              // 系统自动生成
  openid: String,           // 微信 openid，唯一索引
  nickName: String,         // 微信昵称
  avatarUrl: String,        // 微信头像
  realName: String,         // 真实姓名
  department: String,       // 部门/班级
  phone: String,            // 手机号
  role: String,             // 角色：user/admin/superAdmin
  status: Number,           // 0-正常 1-禁用 2-待审核
  createdAt: Date,
  updatedAt: Date
}
```

**rooms 集合**：

```javascript
{
  _id: String,
  name: String,             // 会议室名称
  location: String,         // 位置
  capacity: Number,         // 容纳人数
  facilities: Array,        // 设备清单
  images: Array,            // 图片 URL 数组
  openTime: String,         // 开放时间
  closeTime: String,        // 关闭时间
  minBookingHours: Number,  // 最少预约时长
  maxBookingHours: Number,  // 最大预约时长
  status: Number,           // 0-正常 1-维护中 2-停用
  createdAt: Date,
  updatedAt: Date
}
```

**bookings 集合**：

```javascript
{
  _id: String,
  userId: String,           // 关联 users._id
  roomId: String,           // 关联 rooms._id
  date: String,             // 预约日期（YYYY-MM-DD）
  startTime: String,        // 开始时间（HH:MM）
  endTime: String,          // 结束时间（HH:MM）
  purpose: String,          // 用途描述
  attendees: Number,        // 参与人数
  contactPhone: String,     // 联系人电话
  status: String,           // pending/approved/rejected/cancelled/completed
  needApproval: Boolean,    // 是否需要审批
  approverId: String,       // 审批人 ID
  approveTime: Date,        // 审批时间
  rejectReason: String,     // 拒绝原因
  createdAt: Date,
  updatedAt: Date
}
```

## 六、版本管理规范

### 6.1 Git 提交信息规范

#### 6.1.1 格式

```
<类型>(<范围>): <简要描述>

<正文（可选）>

<脚注（可选）>
```

#### 6.1.2 类型说明

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| `feat` | 新功能 | 添加新功能、新页面、新接口 |
| `fix` | 修复 Bug | 修复问题、错误处理 |
| `docs` | 文档更新 | 更新 README、注释、文档 |
| `style` | 代码格式 | 格式化、缩进、分号等 |
| `refactor` | 重构 | 代码重构，不添加功能也不修复 Bug |
| `perf` | 性能优化 | 提升性能的修改 |
| `test` | 测试相关 | 添加或修改测试代码 |
| `chore` | 杂项 | 构建配置、依赖更新等 |

#### 6.1.3 示例

```bash
# 新功能
feat(预约): 添加时间段预约功能

# Bug 修复
fix(登录): 修复微信登录回调失败问题

# 文档更新
docs: 更新 API 接口文档

# 代码重构
refactor(用户模块): 重构用户认证逻辑
```

### 6.2 自动化提交流程

当需要提交代码时，执行以下步骤：

#### 步骤 1: 检查当前状态

```bash
git status
```

#### 步骤 2: 分析变更内容

根据 git status 的输出，判断：
- 修改了哪些文件
- 变更是什么类型（新功能、Bug 修复、文档更新等）

#### 步骤 3: 生成提交信息

根据变更类型，使用以下格式生成提交信息：
- 新功能: `feat(模块): 描述`
- Bug 修复: `fix(模块): 描述`
- 文档更新: `docs: 描述`
- 代码重构: `refactor(模块): 描述`
- 性能优化: `perf(模块): 描述`
- 测试相关: `test(模块): 描述`
- 其他杂项: `chore: 描述`

#### 步骤 4: 执行提交

```bash
git add .
git commit -m "生成的提交信息"
```

#### 步骤 5: 推送到远程

```bash
git push origin main
```

#### 步骤 6: 验证结果

```bash
git status
git log --oneline -3
```

### 6.3 重要规则

1. **提交前必须检查状态**：不要盲提交，先执行 `git status` 查看变更
2. **提交信息必须规范**：使用中文提交信息，格式为 `类型(范围): 描述`
3. **推送前确认提交成功**：只有提交成功后才能推送
4. **验证最终结果**：推送后检查状态确认成功

### 6.4 常用命令速查

| 操作 | 命令 |
|------|------|
| 查看状态 | `git status` |
| 添加所有文件 | `git add .` |
| 提交 | `git commit -m "类型: 描述"` |
| 推送 | `git push origin main` |
| 查看日志 | `git log --oneline -5` |
| 查看远程地址 | `git remote -v` |
